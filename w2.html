Why DFT, not FFT?

Understanding the difference between a transform and an algorithm

FFT is not a transform ‚Äî it's an algorithm.

The word transform implies a mathematical operation with a precise definition: take an input, apply a formula, get a defined output. That's the DFT. FFT is a clever set of computational tricks that computes the exact same result as DFT ‚Äî just much faster.

DFT ‚Äî The Definition

A mathematical formula that defines what is being computed: for every frequency k, sum up how much of that frequency is present in the signal.

FFT ‚Äî The Algorithm

A recursive, divide-and-conquer method that computes the same DFT output in O(N log N) instead of O(N¬≤). The result is identical.

Same output, different speed. np.fft.fft2() runs the FFT algorithm internally but produces the exact DFT result. There is no mathematical difference in what you get ‚Äî only in how fast you get it.

Long multiplication and a calculator both give you the same product. You learn long multiplication to understand what multiplication means. You use the calculator because it's fast. DFT is the long multiplication ‚Äî FFT is the calculator.

Why does the guide say "DFT converts values into frequency components"?

Because the DFT formula is what defines the conversion. It shows you the math: what each variable means, why complex exponentials are involved, what frequency component k represents. Saying "FFT converts values into frequencies" would be like saying "bubble sort defines what sorting means" ‚Äî it conflates a process with its definition.

Once the concept is clear, the guide uses np.fft.fft2() in all code ‚Äî because in practice you always use the fast algorithm.
